# Plan de iteraciones de Nots26

## 1. Objetivo del plan

Definir una ruta de desarrollo incremental para Nots26, priorizando una versión mínima viable (MVP) y permitiendo la incorporación progresiva de nuevas funcionalidades conforme evolucione el producto.

## 2. Enfoque metodológico

Se utilizará una aproximación basada en Scrum ajustado, con iteraciones cortas y entregables funcionales. Cada iteración deberá producir un incremento verificable del producto, con foco en valor para el usuario y capacidad de adaptación a cambios.

## 3. Principios de trabajo

- Entregar valor de forma incremental.
- Priorizar funcionalidades esenciales para el MVP.
- Revisar resultados frecuentemente y ajustar según feedback.
- Mantener el producto preparado para crecer con nuevas capacidades.
- Separar claramente lo que es esencial de lo que es opcional o futuro.

## 4. Definición de la versión MVP

La versión MVP debe permitir al usuario:

- Crear notas.
- Editar notas.
- Mover notas en el espacio de trabajo.
- Conectar notas entre sí.
- Guardar el estado de forma persistente.
- Visualizar las notas y relaciones de manera clara.

## 5. Iteraciones propuestas

### Iteración 1: Fundación del producto

**Objetivo:** establecer la base visual y funcional del sistema.

**Funcionalidades:**

- Crear estructura inicial de la aplicación.
- Definir espacio de trabajo principal.
- Implementar creación de notas básicas.
- Permitir edición simple del contenido.
- Guardado básico de datos.

**Criterio de salida:**

- El usuario puede crear y editar al menos una nota.
- La aplicación conserva el estado tras recargar.

### Iteración 2: Interacción visual y organización

**Objetivo:** mejorar la experiencia de uso mediante interacción directa.

**Funcionalidades:**

- Implementar arrastre de notas.
- Ajustar posición visual en el canvas.
- Mejorar la experiencia de edición.
- Añadir estilos básicos de color o categoría.

**Criterio de salida:**

- El usuario puede mover notas libremente dentro del espacio de trabajo.
- La posición se conserva correctamente.

### Iteración 3: Conexiones entre notas

**Objetivo:** introducir relaciones entre elementos.

**Funcionalidades:**

- Crear conexiones entre dos notas.
- Visualizar la relación mediante líneas o enlaces.
- Permitir eliminar conexiones.
- Mantener coherencia al editar o borrar notas relacionadas.

**Criterio de salida:**

- El usuario puede vincular dos notas y ver la conexión en la interfaz.

### Iteración 4: Organización y recuperación

**Objetivo:** hacer el producto más útil para el manejo de múltiples notas.

**Funcionalidades:**

- Añadir búsqueda por contenido o título.
- Implementar etiquetas o categorías básicas.
- Filtrar notas por tipo o etiqueta.
- Mejorar la navegación en espacios con varias notas.

**Criterio de salida:**

- El usuario puede encontrar notas rápidamente mediante búsqueda o filtrado.

### Iteración 5: Refinamiento y estabilidad

**Objetivo:** asegurar una experiencia más robusta y preparada para crecer.

**Funcionalidades:**

- Historial básico de cambios.
- Deshacer y rehacer acciones simples.
- Mejoras de rendimiento y estabilidad.
- Ajustes de accesibilidad y diseño visual.

**Criterio de salida:**

- El producto es usable de forma consistente y con menos errores.

## 6. Priorización general

### Alta prioridad

- Creación y edición de notas.
- Arrastre de notas.
- Conexiones entre notas.
- Persistencia de datos.

### Media prioridad

- Etiquetas y filtrado.
- Búsqueda.
- Historial de cambios.
- Deshacer y rehacer.

### Baja prioridad / futuras iteraciones

- Colaboración.
- Importación y exportación avanzada.
- Personalización visual extensa.
- Vistas alternativas del workspace.

## 7. Criterios de éxito del plan

El plan será exitoso si:

- Se entrega una versión MVP funcional en pocas iteraciones.
- Cada iteración aporta valor observable.
- El producto puede adaptarse a nuevas necesidades sin reestructurarse por completo.
- El desarrollo permanece ordenado, medible y escalable.

## 8. Siguiente paso

El siguiente paso recomendado es convertir este plan en una hoja de ruta más detallada, separando por historias de usuario, tareas técnicas y criterios de aceptación por iteración.
